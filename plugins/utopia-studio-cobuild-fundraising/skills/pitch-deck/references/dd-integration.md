# DD Integration Guide

When a Technical Due Diligence report is available for the company, use it to strengthen the pitch deck with evidence-backed claims. This guide explains what to extract and where it goes.

---

## Finding DD Reports

DD reports may be:
- Uploaded directly by the user alongside other materials
- Referenced in conversation ("we already ran DD on this company")
- Available in the uploads directory as a .docx file

If the user mentions DD has been done, ask if they can provide the report.

---

## What to Extract

### For the Solution / Product Slides

- **Architecture assessment** — is the tech well-built? Use positive findings as credibility signals.
- **Scalability evaluation** — can it handle growth? Strong findings support the traction narrative.
- **Technology differentiation** — what's genuinely novel vs. commodity? Strengthens the competitive positioning.
- **IP / defensibility** — proprietary technology, patents, data assets. Feeds into the Growth Flywheel moats.

### For the Team Slide

- **Technical leadership assessment** — DD often evaluates the CTO and engineering team. Positive findings add credibility.
- **Team gaps identified** — mention these in speaker notes as "known and being addressed with this round."

### For the Risk / Speaker Notes

- **Risk flags** — do NOT put these on slides. Include them in speaker notes so the presenter can address them if asked.
- **Evidence ratings** — if the DD report rated claims as SUPPORTED / PARTIAL / CLAIMED / UNSUPPORTED, use SUPPORTED items on slides and note others in speaker notes.

### For the Flywheel Slide

- **Data moat evidence** — DD reports often assess data assets and their defensibility
- **Technical switching costs** — integration depth, API dependencies, training data lock-in
- **Scale economics** — infrastructure efficiency, compute cost trajectories

---

## How to Reference DD Findings

On slides:
- "Independent technical assessment confirms [specific finding]"
- "Architecture reviewed and validated for [scale target]"
- Do NOT reproduce the full DD report — reference it as supporting evidence

In speaker notes:
- "Technical DD flagged [risk area] — mitigation plan is [X]"
- "DD evidence rating for [claim]: [rating] — [brief context]"

---

## What NOT to Do

- Don't put risk flags or negative findings on slides — these go in speaker notes only
- Don't reference the DD report if it's confidential to The Studio and the deck is going to the founder
- Don't overweight DD findings — they're one input, not the whole story
- Don't fabricate DD findings — if no report exists, don't reference one
